# Accessibility Documentation

## Overview

This document explains the accessibility features I implemented in AirBrB. I tried to make the application usable for people with disabilities, including those who use screen readers or navigate with keyboards only. I aimed for WCAG 2.1 Level AA compliance, which is the standard for most websites.

## Color Contrast

One of the first things I checked was whether my text colors have sufficient contrast against the background. I used the WebAIM Contrast Checker to verify all color combinations.

**Primary text (#484848 on white):**
- Contrast ratio: 9.03:1
- This exceeds the WCAG AA requirement of 4.5:1 for normal text

**Secondary text (#767676 on white):**
- Contrast ratio: 4.69:1
- Just meets the 4.5:1 requirement

**Primary button (white text on #FF385C background):**
- Contrast ratio: 5.32:1
- Exceeds requirement

I defined these colors in `frontend/src/theme.js` (lines 21-24). All text in the application should be readable, even for users with visual impairments.

## Image Alternative Text

All listing images have alt attributes. For example, in the ListingCard component:

```javascript
<CardMedia
  component="img"
  image={listing.thumbnail}
  alt={listing.title}
/>
```

This means screen readers will announce the listing title when they encounter the image. For placeholder images, I use descriptive alt text like "No Image Available".

See `frontend/src/components/listing/ListingCard.jsx` (line 49).

## Text Sizing

I used rem units for all font sizes instead of px. This is important because rem units scale with the user's browser settings. If someone has set their browser to use larger text, my application will respect that.

Users can also zoom in/out (I tested up to 200% zoom) and the layout doesn't break.

Typography configuration: `frontend/src/theme.js` (lines 36-65)

## Not Relying on Color Alone

I made sure information isn't conveyed by color alone:

**Star ratings:** Show both the star icon AND a numeric value (e.g., "4.5")

**Notifications:** Each notification has:
- A color (blue, green, or red)
- An icon
- Text label (e.g., "Booking Request", "Booking Accepted")

**Form errors:** Display both:
- Red color
- Error icon
- Text message explaining the problem

This way, users who are colorblind or using screen readers still get all the information.

## Keyboard Navigation

I spent a fair amount of time testing keyboard navigation. The entire application can be used without a mouse:

**Tab key:** Move forward through interactive elements
**Shift+Tab:** Move backward
**Enter:** Activate buttons, submit forms, open links
**Space:** Activate buttons and checkboxes
**Escape:** Close modals and menus

**Focus indicators:** Material-UI provides default focus outlines on all interactive elements. I kept these because they're important for keyboard users to see where they are on the page.

**Focus order:** Elements receive focus in a logical order (left to right, top to bottom), matching the visual layout.

I tested by unplugging my mouse and navigating through the entire application. Everything worked.

## Form Accessibility

All form fields have associated labels. For example:

```javascript
<TextField
  label="Email"
  type="email"
  autoComplete="email"
  required
  fullWidth
/>
```

Material-UI's TextField component automatically associates the label with the input, which is crucial for screen readers. The label is announced before the input field, so users know what to enter.

**Required fields:** Marked with an asterisk (*)

**Error messages:** Associated with their fields and announced by screen readers

**Auto-complete attributes:** Help browsers and assistive technology suggest appropriate values

I tested forms with screen readers (NVDA on Windows) and they work well.

See the Login and Register pages for examples.

## Semantic HTML

I used proper HTML structure:
- Navigation is wrapped in appropriate containers (Material-UI's AppBar)
- Headings follow a logical hierarchy (h1, then h2, then h3, etc.)
- Lists use proper list elements
- Buttons are actually button elements, not divs with click handlers

Material-UI components render semantic HTML by default, which helped a lot.

## Clear Error Messages

When something goes wrong, I try to explain what happened and how to fix it. For example:

```javascript
if (password !== confirmPassword) {
  setError('Passwords do not match');
  return;
}
```

The error message:
- Tells the user exactly what's wrong
- Persists until they fix it
- Is associated with the relevant form field
- Uses both color and text (not color alone)

## ARIA Attributes

Material-UI adds most ARIA attributes automatically, but I added some manually where needed. For example, the notification bell:

```javascript
<IconButton
  aria-label="Open notifications"
  onClick={handleMenuOpen}
>
  <Badge badgeContent={unreadCount}>
    <NotificationsIcon />
  </Badge>
</IconButton>
```

The `aria-label` tells screen readers what the button does. Without it, a screen reader would just announce "button" with no context.

Similarly, menus have `aria-labelledby` to associate them with their trigger buttons.

See `frontend/src/components/notifications/NotificationPanel.jsx` for more examples.

## Responsive Design Accessibility

**Touch targets:** All interactive elements are at least 44x44px, which is the minimum recommended size for touch. This is Material-UI's default.

**Zoom enabled:** I didn't set a maximum scale on the viewport, so users can pinch-to-zoom on mobile devices.

**Mobile menu:** The hamburger menu is keyboard accessible and works with screen readers.

## Known Limitations

There are some things I didn't have time to implement:

1. **Skip navigation link:** Keyboard users have to tab through the entire navbar to get to main content. I could add a "Skip to main content" link that appears on focus.

2. **ARIA live regions:** Real-time updates (like new notifications) aren't announced automatically. I'd need to add `aria-live` regions for this.

3. **High contrast mode:** The application relies on system settings. I could add a custom high contrast theme.

4. **File input:** The image upload for listings could be more accessible with better labeling and instructions.

## Future Improvements

## Conclusion

I tried to build accessibility into the application from the start rather than adding it as an afterthought. Using Material-UI helped a lot because it has good default accessibility. The main things I focused on were:
- Sufficient color contrast
- Keyboard navigation
- Screen reader compatibility
- Semantic HTML
- Clear error messages

There's still room for improvement, but I believe the application is usable by people with diverse abilities.
